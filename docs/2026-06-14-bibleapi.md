# Technical Design: Keyword-to-Prayer Bible Passage Search
## Context: Prayer Rhythms · Prayer Wall

> **Repo context:** This document is written for the `prayer-wall` repository.
> Stack: React + TypeScript + Vite (frontend), Supabase (Postgres + Edge Functions + Storage), Deno (edge runtime), Resend (email delivery).
> The Prayer Wall is a product under the **Prayer Rhythms** umbrella (`prayerrhythms.org`).
> All Bible API calls will run inside Supabase Edge Functions — **not** a Python/Flask backend.

# Technical Design: Keyword-to-Prayer Bible Passage Search Using NIV-Compatible API Sources

## 1. Objective

Build an application service that accepts one or more user-supplied keywords, themes, or needs such as `fear`, `healing`, `forgiveness`, `anxiety`, `gratitude`, or `guidance`, and returns prayer-relevant Bible passages. The service should support NIV display when licensing allows it, but should not depend on copying, storing, embedding, or redistributing NIV text outside the permitted API and licensing terms.

The recommended design is **not**:

```text
User keyword → direct Bible API search → return whatever matches
```

The recommended design is:

```text
User keyword/theme
  → normalized intent model
  → internal prayer-passage index
  → ranked passage references
  → licensed Bible API fetch by reference
  → response with passage, metadata, prayer prompt, attribution
```

This separation matters because most Bible APIs are good at retrieving Scripture text by reference, but they are not usually designed to understand pastoral categories like prayer, lament, confession, praise, intercession, fear, grief, or spiritual encouragement.

## 2. Recommended API Providers

### 2.1 YouVersion Platform API

YouVersion is currently the most attractive option for an NIV-backed application because its public developer documentation lists a REST API, a production base URL, and the `X-YVP-App-Key` header for authentication. Their quick reference lists NIV as Bible ID `111`.

Relevant implementation facts from the YouVersion documentation:

- Production base URL: `https://api.youversion.com/v1`
- Authentication header: `X-YVP-App-Key`
- Available Bible versions can be requested through `/bibles`
- Specific Bible details can be requested through `/bibles/{version_id}`
- Books can be requested through `/bibles/{version_id}/books`
- Chapter verses can be requested through `/bibles/{version_id}/books/{book_usfm}/chapters/{chapter_number}/verses`
- NIV appears in the common Bible IDs table as ID `111`
- Their docs recommend caching, error handling, timeouts, validation, and respecting rate-limit headers

Source: YouVersion Platform Developer Docs, Quick Reference  
https://developers.youversion.com/quick-reference

### 2.2 API.Bible

API.Bible is another viable provider, especially for non-commercial usage or when the desired translation is available under the selected plan. Their documentation describes API key authentication through the `api-key` header, v1 endpoints, and a published rate limit for non-commercial usage.

Relevant implementation facts from API.Bible documentation:

- API key is sent using the `api-key` header
- Current API version is `v1`
- Non-commercial usage is documented as free
- Rate limit is documented as 5,000 queries per day
- Up to 500 consecutive verses may be requested at a time
- Commercial usage or increased access requires contacting API.Bible

Source: API.Bible Getting Started documentation  
https://docs.api.bible/api-reference/getting-started/

API.Bible’s public site also states that not all Bible versions are available for all use cases and specifically notes that NIV commercial use is not available through their listed terms.

Source: API.Bible product site  
https://api.bible/

### 2.3 Biblica Licensing Constraint

NIV is stewarded by Biblica, so licensing needs to be treated as a first-class architectural concern, not a legal footnote.

Biblica’s permissions page says use through API.Bible or YouVersion may qualify under Express Licensing only when both conditions are true:

1. The application is truly non-commercial.
2. The application does not include artificial intelligence or machine learning features or functionality.

Biblica also states that written permission is required when use falls outside the general guidelines.

Source: Biblica Permissions  
https://www.biblica.com/permissions/

This means a keyword-to-prayer feature can become licensing-sensitive depending on implementation. A simple curated lookup table is much safer than using AI/ML to generate, classify, summarize, embed, or transform NIV text.

## 3. High-Level Architecture

In this repo, there is no separate backend server. All server-side logic runs as **Supabase Edge Functions** (Deno/TypeScript). The frontend is React + Vite, communicating with Supabase directly via the anon key or through edge function endpoints.

```text
  React (Vite)                   Supabase Edge Functions (Deno)
  ─────────────────              ──────────────────────────────────────
  WallPage / AdminPage           send-reminders  (pg_cron → hourly)
       │                              │
       │  supabase-js (anon key)      │  supabase-js (service role key)
       ▼                              ▼
  prayer_wall schema (Postgres)
  ─────────────────────────────────────────────────────────────────────
  organizations  walls  commitments  prayer_points  commitment_categories
  email_rhythms  commitment_rhythms  email_logs  wall_theme
  message_categories
  ── NEW ──
  prayer_themes  prayer_keywords  prayer_passages  prayer_passage_themes
  bible_api_cache
       │
       ▼
  Bible Provider (called from edge function, never from browser)
  YouVersion API  OR  API.Bible
       │
       ▼
  NIV passage text → inserted into email HTML → sent via Resend
```

**Key constraint:** Bible API keys (`YOUVERSION_APP_KEY`, `API_BIBLE_KEY`) are Supabase secrets. They are never exposed to the browser. All provider calls go through edge functions.

## 4. Why an Internal Prayer Index Is Necessary

A Bible text search endpoint can find literal keyword matches. For example, searching `fear` may return passages that contain the word "fear." But a prayer-oriented system usually needs semantic and pastoral mapping:

| User Need | Likely Useful Passages | Reason |
|---|---|---|
| anxiety | Philippians 4:6-7, Matthew 6:25-34, 1 Peter 5:7 | Directly addresses worry, prayer, care, and peace |
| forgiveness | Psalm 51, Matthew 6:12, 1 John 1:9 | Confession, repentance, assurance |
| grief | Psalm 34:18, Matthew 5:4, John 11:35 | Comfort, mourning, presence of God |
| guidance | Proverbs 3:5-6, James 1:5, Psalm 25 | Wisdom, trust, direction |
| gratitude | 1 Thessalonians 5:16-18, Psalm 100, Colossians 3:15-17 | Thanksgiving and praise |

A literal API search may miss `anxiety → peace`, `grief → comfort`, `guidance → wisdom`, or `forgiveness → confession`. A curated index solves this.

## 5. Data Model

All tables live in the `prayer_wall` Postgres schema, consistent with the existing schema in `supabase/migrations/001_initial_schema.sql` through `012_pg_cron_schedule.sql`. PKs are UUIDs (`gen_random_uuid()`), not serials. The existing schema already defines `organizations`, `walls`, `commitments`, `prayer_points`, `email_rhythms`, `commitment_rhythms`, and `email_logs`. The new tables below extend it.

The `VITE_ORG_ID` and `VITE_WALL_ID` env vars (see `.env.example`) scope all data — every new table that is wall- or org-scoped follows the same pattern.

### 5.1 New Tables (migration `013_bible_search.sql`)

```sql
-- ── Bible passage index ─────────────────────────────────────

CREATE TABLE prayer_wall.prayer_themes (

  slug         TEXT NOT NULL UNIQUE,
  display_name TEXT NOT NULL,

CREATE TABLE prayer_wall.prayer_passages (

  book_usfm        TEXT NOT NULL,          -- e.g. 'PHP'
  chapter_start    SMALLINT NOT NULL,
  verse_start      SMALLINT NOT NULL,
  chapter_end      SMALLINT NOT NULL,
  verse_end        SMALLINT NOT NULL,
  canonical_weight NUMERIC(5,2) DEFAULT 1.0,
  is_active        BOOLEAN NOT NULL DEFAULT true,
  created_at       TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE prayer_wall.prayer_passage_themes (
  passage_id       UUID NOT NULL REFERENCES prayer_wall.prayer_passages(id) ON DELETE CASCADE,
  theme_id         UUID NOT NULL REFERENCES prayer_wall.prayer_themes(id) ON DELETE CASCADE,
  relevance_weight NUMERIC(5,2) DEFAULT 1.0,
  PRIMARY KEY (passage_id, theme_id)
);

CREATE TABLE prayer_wall.prayer_keywords (
  id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  keyword            TEXT NOT NULL,
  normalized_keyword TEXT NOT NULL,
  theme_id           UUID REFERENCES prayer_wall.prayer_themes(id) ON DELETE CASCADE,
  weight             NUMERIC(5,2) DEFAULT 1.0,
  created_at         TIMESTAMPTZ DEFAULT now(),
  UNIQUE (normalized_keyword, theme_id)
);

-- ── Short-lived cache for Bible API responses ────────────────
-- Do NOT cache NIV text permanently. Use short expires_at TTLs.

CREATE TABLE prayer_wall.bible_api_cache (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  provider         TEXT NOT NULL,          -- 'youversion' | 'api.bible'
  bible_id         TEXT NOT NULL,          -- '111' for NIV
  reference        TEXT NOT NULL,
  response_json    JSONB NOT NULL,
  expires_at       TIMESTAMPTZ NOT NULL,
  created_at       TIMESTAMPTZ DEFAULT now(),
  UNIQUE (provider, bible_id, reference)
);

CREATE INDEX ix_bible_api_cache_expires ON prayer_wall.bible_api_cache(expires_at);
```

### 5.2 Relationship to Existing Tables

The passage search integrates with existing tables without modifying them:

- **`commitments`** — has `prayer_request` (free text) and links to `prayer_points` (discrete items). Both are keyword sources for matching themes.
- **`prayer_points`** — `body` field is the primary source for per-warrior theme matching in `send-reminders`.
- **`commitment_rhythms`** — already links warriors to rhythm schedules; the email function uses this to know which warriors to email and when.
- **`email_logs`** — already tracks `wall_id`, `commitment_id`, `status`, `resend_message_id`; no changes needed.

Important licensing note: caching provider API responses may be restricted. For NIV, avoid long-term storage of verse text unless your license explicitly allows it. Prefer short TTL caching (minutes to hours), reference-only persistence, and always render copyright attribution.

## 7. Provider Abstraction

All provider calls run in **Deno (TypeScript)** inside a Supabase Edge Function. The shared adapter file lives at `supabase/functions/_shared/bible-provider.ts` and is imported by both `send-reminders` and a future `bible-search` edge function.

Provider API keys are Supabase secrets — set via `supabase secrets set` and accessed via `Deno.env.get()`. Never expose them to the React frontend.

```typescript
// supabase/functions/_shared/bible-provider.ts
import type { BiblePassageText, PassageReference } from "./bible-types.ts";

export interface BibleProvider {
  getPassage(bibleId: string, reference: PassageReference): Promise<BiblePassageText>;
}
```

### 7.1 YouVersion Adapter (Deno)

Secret: `YOUVERSION_APP_KEY` (set via `supabase secrets set YOUVERSION_APP_KEY=...`)

```typescript
// supabase/functions/_shared/youversion-provider.ts
import type { BibleProvider, BiblePassageText, PassageReference } from "./bible-provider.ts";

const BASE_URL = "https://api.youversion.com/v1";

export class YouVersionProvider implements BibleProvider {
  private readonly appKey: string;

  constructor() {
    this.appKey = Deno.env.get("YOUVERSION_APP_KEY") ?? "";
  }

  private headers(): Record<string, string> {
    return { "X-YVP-App-Key": this.appKey, "Accept": "application/json" };
  }

  async getPassage(bibleId: string, ref: PassageReference): Promise<BiblePassageText> {
    const url = `${BASE_URL}/bibles/${encodeURIComponent(bibleId)}/books/` +
      `${encodeURIComponent(ref.bookUsfm)}/chapters/${ref.chapterStart}/verses`;

    const res = await fetch(url, { headers: this.headers() });
    if (!res.ok) throw new Error(`YouVersion ${res.status}: ${await res.text()}`);
    const payload = await res.json() as { data?: Array<{ verse?: number; content?: string }> };

    const verses = (payload.data ?? [])
      .filter((v) => {
        const n = Number(v.verse);
        return n >= ref.verseStart && n <= ref.verseEnd;
      })
      .map((v) => (v.content ?? "").trim())
      .filter(Boolean);

    return {
      reference: ref.reference,
      translation: bibleId === "111" ? "NIV" : bibleId,
      provider: "youversion",
      text: verses.join("\n"),
      copyright: null,  // extract from payload.meta if available
    };
  }
}
```

### 7.2 API.Bible Adapter (Deno)

Secret: `API_BIBLE_KEY` (set via `supabase secrets set API_BIBLE_KEY=...`)

```typescript
// supabase/functions/_shared/apibible-provider.ts
import type { BibleProvider, BiblePassageText, PassageReference } from "./bible-provider.ts";

const BASE_URL = "https://api.scripture.api.bible/v1";

export class ApiBibleProvider implements BibleProvider {
  private readonly apiKey: string;

  constructor() {
    this.apiKey = Deno.env.get("API_BIBLE_KEY") ?? "";
  }

  async getPassage(bibleId: string, ref: PassageReference): Promise<BiblePassageText> {
    const url = `${BASE_URL}/bibles/${bibleId}/passages/${encodeURIComponent(ref.reference)}`;
    const res = await fetch(url, {
      headers: { "api-key": this.apiKey, "Accept": "application/json" },
    });
    if (!res.ok) throw new Error(`API.Bible ${res.status}: ${await res.text()}`);
    const payload = await res.json() as { data?: { reference?: string; content?: string; copyright?: string } };
    const data = payload.data ?? {};

    return {
      reference: data.reference ?? ref.reference,
      translation: bibleId,
      provider: "api.bible",
      text: data.content ?? "",
      copyright: data.copyright ?? null,
    };
  }
}
```

## 8. Repository Layer

In this repo, "repository" means Supabase queries using the typed `supabase-js` client — the same pattern used in `WarriorsAdmin.tsx`, `RhythmsAdmin.tsx`, and `send-reminders/index.ts`. There is no SQLAlchemy or ORM. The edge function uses the **service role key** to bypass RLS.

```typescript
// supabase/functions/_shared/prayer-search-repo.ts
import { createClient } from "jsr:@supabase/supabase-js@2";

const supabase = createClient(
  Deno.env.get("SUPABASE_URL")!,
  Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!,
  { db: { schema: "prayer_wall" } },  // matches client.ts in the frontend
);

export async function findThemesByTerms(terms: string[]): Promise<string[]> {
  const normalized = terms.map((t) => t.toLowerCase().trim()).filter(Boolean);
  const { data } = await supabase
    .from("prayer_keywords")
    .select("theme_id")
    .in("normalized_keyword", normalized);
  return [...new Set((data ?? []).map((r: { theme_id: string }) => r.theme_id))];
}

export async function findPassagesForThemes(
  themeIds: string[],
  limit = 5,
): Promise<Array<{ reference: string; bookUsfm: string; chapterStart: number; verseStart: number; chapterEnd: number; verseEnd: number; score: number }>> {
  // Fetch passage-theme links, join to passages, aggregate score in JS
  const { data: links } = await supabase
    .from("prayer_passage_themes")
    .select("passage_id, relevance_weight, prayer_passages(id, reference, book_usfm, chapter_start, verse_start, chapter_end, verse_end, canonical_weight, is_active)")
    .in("theme_id", themeIds);

  type Link = { passage_id: string; relevance_weight: number; prayer_passages: { id: string; reference: string; book_usfm: string; chapter_start: number; verse_start: number; chapter_end: number; verse_end: number; canonical_weight: number; is_active: boolean } | null };

  const scoreMap = new Map<string, { passage: NonNullable<Link["prayer_passages"]>; score: number }>();
  for (const link of (links ?? []) as Link[]) {
    const p = link.prayer_passages;
    if (!p || !p.is_active) continue;
    const prev = scoreMap.get(p.id);
    const score = link.relevance_weight * p.canonical_weight;
    scoreMap.set(p.id, { passage: p, score: (prev?.score ?? 0) + score });
  }

  return [...scoreMap.values()]
    .sort((a, b) => b.score - a.score)
    .slice(0, limit)
    .map(({ passage: p, score }) => ({
      reference: p.reference,
      bookUsfm: p.book_usfm,
      chapterStart: p.chapter_start,
      verseStart: p.verse_start,
      chapterEnd: p.chapter_end,
      verseEnd: p.verse_end,
      score,
    }));
}
```

## 9. Service Layer

The service logic lives inside a Supabase Edge Function. In this repo the first integration point is `send-reminders`, which already handles the full warrior→email loop. The service function below is extracted logic that can be called from `send-reminders` or a future `bible-search` edge function.

```typescript
// supabase/functions/_shared/prayer-search-service.ts
import type { BibleProvider } from "./bible-provider.ts";
import { findThemesByTerms, findPassagesForThemes } from "./prayer-search-repo.ts";

function normalizeTerms(input: string): string[] {
  return [...new Set(
    input.toLowerCase().split(/[\s,;:/|]+/).map((t) => t.trim()).filter(Boolean),
  )];
}

export async function findPassageForText(
  text: string,
  provider: BibleProvider,
  bibleId = "111",
): Promise<{ reference: string; translation: string; text: string; copyright: string | null } | null> {
  const terms = normalizeTerms(text);
  if (terms.length === 0) return null;

  const themeIds = await findThemesByTerms(terms);
  if (themeIds.length === 0) return null;

  const passages = await findPassagesForThemes(themeIds, 1);
  if (passages.length === 0) return null;

  const top = passages[0];
  try {
    const result = await provider.getPassage(bibleId, {
      reference: top.reference,
      bookUsfm: top.bookUsfm,
      chapterStart: top.chapterStart,
      verseStart: top.verseStart,
      chapterEnd: top.chapterEnd,
      verseEnd: top.verseEnd,
    });
    return result;
  } catch (err) {
    console.error(`Bible provider error for ${top.reference}:`, err);
    return null;  // graceful degradation — email sends without a passage
  }
}
```

## 11. Ranking Strategy

A simple first version can rank by deterministic weights:

```text
score =
  theme_match_weight
  + keyword_weight
  + passage_theme_relevance_weight
  + canonical_weight
  + recency_boost_or_popularity_boost
```

Recommended ranking dimensions:

| Dimension | Purpose |
|---|---|
| keyword exact match | Boost direct matches like `fear`, `healing`, `thanksgiving` |
| theme expansion | Map related words like `anxiety → peace, worry, trust` |
| passage relevance | Prefer passages explicitly connected to prayer |
| canonical weight | Let editors promote especially useful passages |
| diversity penalty | Avoid returning five passages from the same chapter |
| user context | Optional, only if licensing/privacy allows |

Avoid ML embeddings over NIV text unless you have explicit permission for AI/ML use. If semantic matching is desired, embed your own metadata, not the NIV text.

Acceptable low-risk embedding candidate:

```json
{
  "reference": "Philippians 4:6-7",
  "themes": ["anxiety", "peace", "prayer", "thanksgiving"],
  "summary": "A passage encouraging prayer and thanksgiving instead of worry."
}
```

Higher-risk embedding candidate:

```json
{
  "reference": "Philippians 4:6-7",
  "niv_text": "..."
}
```

## 12. Caching Strategy

### 12.1 Cache What Is Safe

Prefer caching:

- Bible version metadata
- Book metadata
- Chapter metadata
- Passage references
- Theme mappings
- Search results that contain references but not copyrighted text
- Provider response ETags or last-modified headers, if available
- Short-lived rendered responses if license permits

### 12.2 Be Careful Caching NIV Text

For NIV, do not assume you can store full passage text in your database indefinitely. A safer default is:

```text
Cache NIV API responses in Redis for minutes or hours, not permanently.
Persist only references, metadata, and your own commentary/prompt content.
Render NIV text only after fetching it from the licensed provider.
```

Example Redis cache key:

```text
bible:v1:youversion:111:Philippians%204%3A6-7
```

Example TTL policy:

| Content | Suggested TTL |
|---|---:|
| provider Bible list | 24 hours |
| book/chapter metadata | 7 days |
| passage response containing NIV text | 15 minutes to 6 hours, depending on license |
| internal theme index | until invalidated |

## 13. Error Handling

### 13.1 Provider Errors

| HTTP Status | Meaning | Handling |
|---:|---|---|
| 400 | Bad request or malformed reference | Validate reference before provider call |
| 401 | Missing/invalid API key | Log securely, return generic provider unavailable message |
| 403 | Translation not authorized | Fallback to another translation or tell user NIV is unavailable |
| 404 | Reference not found | Suppress result or mark passage unavailable |
| 429 | Rate limited | Exponential backoff, cache, queue retry if async |
| 500+ | Provider unavailable | Return curated references without text if acceptable |

### 13.2 Application-Level Degradation

When the provider is unavailable, the service can still return:

```json
{
  "reference": "Philippians 4:6-7",
  "translation": "NIV",
  "text": null,
  "status": "TEXT_UNAVAILABLE",
  "message": "Passage text is temporarily unavailable."
}
```

This preserves usefulness without violating licensing by serving locally stored copyrighted text.

## 15. Security

### 15.1 Secrets

Store provider credentials in environment variables or a secret manager:

```text
YOUVERSION_APP_KEY
API_BIBLE_KEY
```

Never expose provider keys to the client application. All Bible API calls should go through your backend.

### 15.2 Abuse Protection

Add:

- Per-IP rate limiting
- Per-user rate limiting if authenticated
- Query length limits
- Allowlist for supported translations
- Request timeout controls
- Circuit breaker around provider APIs

### 15.3 Input Validation

Avoid letting users pass arbitrary provider URLs, raw SQL filters, or unsupported Bible IDs.

## 16. Licensing-Safe Design Rules

1. Store references, not NIV text.
2. Store your own metadata, not copied headings or study notes from a provider.
3. Fetch NIV text only from an authorized provider.
4. Render required copyright attribution.
5. Avoid AI/ML over NIV text unless explicitly licensed.
6. Do not scrape Bible Gateway, Bible.com, or other public-facing sites.
7. Confirm whether the application is commercial, fundraising-related, nonprofit, internal, or public.
8. Treat donor-supported, ad-supported, or subscription-supported projects as potentially commercial.
9. Keep an implementation switch that disables NIV and falls back to an open translation if access changes.
10. Keep provider terms, permissions approval, and Bible IDs in configuration, not hard-coded business logic.

## 17. Translation Fallback Strategy

Recommended default order:

```text
NIV through YouVersion, if licensed and available
→ API.Bible translation, if licensed and available
→ Berean Standard Bible / World English Bible / ASV or another permitted open or accessible translation
→ reference-only response
```

The fallback response should be explicit:

```json
{
  "requested_translation": "NIV",
  "resolved_translation": "WEBUS",
  "reason": "NIV is not available for this request context."
}
```

## 20. Practical Recommendation for This Repo

**Start with API.Bible** rather than YouVersion — API.Bible's access is self-service (sign up, get a key, start calling). YouVersion's API requires approval and may not be accessible immediately. Use YouVersion once access is granted; the `BibleProvider` interface makes it a one-line swap.

**The v1 deliverable is one passage per reminder email**, not a search UI. Wire `findPassageForText` into `send-reminders` against the warrior's prayer points. This is achievable once migration `013_bible_search.sql` is run and the passage index is seeded.

**Own the prayer intelligence in Postgres** (`prayer_themes`, `prayer_keywords`, `prayer_passages` in the `prayer_wall` schema). These tables are the differentiator — not the Bible API call. The API just fetches text for a reference you already chose.

**The `bible_api_cache` table replaces Redis** for this stack. Supabase Postgres is already the data store; adding Redis is unnecessary overhead. Short-TTL rows with a cleanup job or `expires_at` filter are sufficient.

**Track the Prayer Rhythms product boundary**: the Prayer Wall is one product. The passage search index, the email enrichment, and the future `bible-search` edge function are all infrastructure that Prayer Rhythms can eventually offer to other walls under `prayerrhythms.org` — which means the `prayer_themes` and `prayer_passages` tables should be **org-scoped or global**, not wall-scoped.

The best immediate next step: get a Supabase secret for `API_BIBLE_KEY`, write `013_bible_search.sql`, seed 20–30 passages for the most common prayer themes, and confirm a passage appears in a test reminder email.
